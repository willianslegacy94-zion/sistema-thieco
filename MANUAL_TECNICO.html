Capítulo 3
Subir / Parar o Sistema
Primeira vez
cp .env.example .env          # preencher senhas
docker compose up -d           # sobe tudo (migrations automáticas)
Operações do dia a dia
docker compose up -d           # subir
docker compose down            # parar (preserva dados)
docker compose logs -f         # logs em tempo real
docker compose logs -f backend # logs de serviço específico
docker compose ps              # status dos containers
Rebuild após atualização de código
git pull origin main

# Com cache (apenas código mudou)
docker compose up -d --build

# Sem cache (package.json ou Dockerfile mudou)
docker compose build --no-cache backend
docker compose build --no-cache frontend
docker compose up -d
Reiniciar serviço específico
docker compose restart backend
docker compose restart frontend
Atenção: docker compose down -v apaga o volume do banco de dados. Use apenas em ambiente de desenvolvimento ou com backup previamente feito.

Capítulo 5
Autenticação e Papéis

Login via POST /api/auth/login — retorna JWT. O token é salvo em localStorage com a chave thieco_auth_token e enviado em todas as requisições como Authorization: Bearer <token>.

Payload do JWT
{
  "id":               1,
  "username":         "thieco",
  "nome":             "Thieco Leandro",
  "role":             "admin",
  "unidade":          "tambore",
  "profissional_id":  1
}
Middlewares — backend/middleware/auth.js
MiddlewareComportamento
authenticateValida Bearer token JWT, injeta req.user com os dados decodificados
requireAdminRetorna 403 se req.user.role !== 'admin'
Resolução de Unidade nas Rotas Protegidas

Padrão adotado em vendas.js e combos.js (rota ativar):

const unidade = req.user.role === 'operador'
  ? req.user.unidade               // operador: fixado no JWT
  : (req.body.unidade ?? req.user.unidade); // admin: body ou fallback JWT
Operadores têm unidade fixada no JWT. Admins podem enviar unidade no body ou usar a própria como fallback — evita que vendas do Tamboré caiam em Mutinga.

SWR — Comportamento de Polling
Configuração em useBarbeariaData.js
  • refreshInterval: 3000 — polling a cada 3 segundos
  • keepPreviousData: true — não pisca ao atualizar
  • dedupingInterval: 2000 — deduplicação de requests
  • Chaves SWR ficam null enquanto datas são inválidas — suspende o fetch
Buffer de Datas no FilterBar

FilterBar.jsx mantém estado local (localInicio, localFim) e só propaga ao estado pai quando isValidDate(v) retorna true. Evita re-renders do Dashboard inteiro a cada keystroke durante digitação manual.

Fluxo de Dados — Registro de Venda
Usuário preenche formulário (RegistroVenda.jsx)
  ↓
AbaVenda.onSubmit():
  ├─ Encontra o barbeiro na lista → extrai barbeiro.unidade
  ├─ Calcula comissao = valor × (percentual_comissao / 100)
  └─ Monta payload: { unidade, profissional_id, servico, valor, comissao, ... }
  ↓
api.criarVenda(payload) → POST /api/vendas
  ↓
Backend (vendas.js):
  ├─ authenticate → req.user injetado
  ├─ express-validator valida campos
  ├─ Resolve unidade: operador usa JWT · admin usa body ?? JWT
  └─ Venda.create() → INSERT INTO vendas RETURNING *
  ↓
useInvalidarDashboard() → invalida cache SWR → Dashboard recarrega

Taxas PagBank e Cálculo de DRE

Taxas armazenadas na tabela configuracoes (chave/valor) — atualizáveis via SQL sem rebuild.

ChaveValorForma de Pagamento
taxa_debito0.0119Débito Visa/Master/Elo — 1,19%
taxa_credito0.0349Crédito à vista — 3,49%
taxa_pix0Pix — isento
taxa_dinheiro0Dinheiro — sem taxa
taxa_cortesia0Cortesia — sem taxa
Fórmula DRE

Faturamento Bruto
  − Taxas maquininha (taxa × valor, por forma de pagamento)
= Faturamento Líquido
  − Comissões dos barbeiros
  − Gastos do período
= Lucro Operacional

Para atualizar uma taxa:

UPDATE configuracoes SET valor = '0.0139' WHERE chave = 'taxa_debito';

Capítulo 11
Troubleshooting
Container não sobe / fica em "starting"
docker compose logs postgres
docker compose logs backend
Causas comuns: .env não existe, senha errada, volume corrompido. Em último caso: docker compose down -v (apaga dados) + docker compose up -d.
Backend retorna 422 ao registrar venda
Verificar: (1) o barbeiro selecionado tem unidade preenchida? (2) forma_pagamento é um dos valores válidos? (3) data no formato YYYY-MM-DD?
docker compose logs backend | grep "422\|erro\|Erro"
Venda aparecendo na unidade errada
A unidade da venda vem de barbeiro.unidade no frontend (RegistroVenda.jsx → AbaVenda.onSubmit). Verificar se o barbeiro está cadastrado com a unidade correta:
SELECT id, nome, unidade FROM profissionais ORDER BY unidade, nome;
Para corrigir: UPDATE profissionais SET unidade = 'tambore' WHERE id = <id>;
Barbeiro não aparece no seletor de vendas
O barbeiro está inativo (ativo = false). O endpoint GET /profissionais retorna apenas ativos. Reative em Gestão de Time → Equipe ou via SQL:
UPDATE profissionais SET ativo = true WHERE id = <id>;
Dashboard travado ao digitar data
FilterBar.jsx usa estado local e isValidDate() como guarda. Se regredir, verificar se os inputs usam value={localInicio} / value={localFim} (não filtros.inicio diretamente).
Rebuild sem efeito — mudanças não aparecem
Docker pode estar usando cache. Forçar rebuild completo:
docker compose build --no-cache frontend
docker compose build --no-cache backend
docker compose up -d
Redefinir senha de usuário
Gerar hash via Node e atualizar no banco:
docker exec -it thieco_api node -e "
const bcrypt = require('bcryptjs');
bcrypt.hash('NovaSenha123!', 12).then(h => console.log(h));
"
# Copiar o hash gerado e:
docker exec -it thieco_db psql -U postgres sistema_thieco -c \
  "UPDATE usuarios SET senha_hash = '<hash>' WHERE username = 'mutinga';"
Ver usuários cadastrados
SELECT id, username, role, unidade_acesso, ativo FROM usuarios;
Capítulo 12
Decisões Técnicas Relevantes
ON DELETE SET NULL em vendas.profissional_id

Ao excluir ou inativar um barbeiro, as vendas históricas são preservadas com profissional_id = NULL. Os relatórios agrupam essas vendas como "Barbeiro Removido". Garante integridade histórica sem bloquear operações.

Soft delete para profissionais (ativo = false)

Barbeiros nunca são deletados do banco. A flag ativo = false os remove de todos os seletores e relatórios sem perder histórico. Resolve cenários de substituição de funcionários e permite reativação.

SWR com polling de 3 segundos

Usado em vez de WebSocket para simplificar a infraestrutura. A latência de 3s é aceitável para o contexto. keepPreviousData: true evita flash de loading a cada poll, mantendo a UX fluida.

Migrations idempotentes no startup

Toda DDL está em backend/models.js → runMigrations() e usa cláusulas IF NOT EXISTS e ON CONFLICT DO NOTHING. O backend pode ser reiniciado a qualquer momento sem risco de migrations destrutivas ou duplicadas.

Unicidade do catálogo por unidade

CREATE UNIQUE INDEX uq_catalogo_nome_unidade ON catalogo (nome, COALESCE(unidade, '')) — permite o mesmo nome de serviço em unidades diferentes (ex: "Corte" R$45 em Mutinga e R$70 em Tamboré). Itens sem unidade (NULL = compartilhados) ainda são únicos pelo nome.
